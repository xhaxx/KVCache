## 1. 引言

### 1.1 编写目的

本文档旨在明确 LightCache 键值缓存中间件的功能需求与非功能需求，作为后续设计、开发、测试、验收的基准依据。

### 1.2 产品范围

LightCache 是面向学习和原型验证场景的单机键值缓存中间件，核心目标是**通过亲手实现来理解缓存中间件的内部机制**，不追求生产级高可用，但要求接口规范、代码可读、可测试。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| CacheEngine | 核心缓存引擎，提供 put/get/del 等原子操作 |
| TTL (Time To Live) | 键的存活时间，单位秒；-1 表示永不过期 |
| 惰性删除 (Lazy Expiration) | 读取时检查键是否过期，过期则删除 |
| 定期删除 (Periodic Expiration) | 后台线程定时扫描并清理过期键 |
| 快照持久化 (Snapshot Persistence) | 将当前全部有效键值对序列化写入磁盘 |
| 命中率 (Hit Rate) | get 操作命中次数 / 总 get 次数 |

- Spring Boot 3.4.3 Reference Documentation

## 2. 产品描述

### 2.1 产品视角

LightCache 作为嵌入式或独立部署的缓存中间件，向业务系统提供键值存取能力。架构上分为三个模块：

- **Core（存储引擎）**：纯 Java 实现，可嵌入任何 JVM 应用
- **Server（服务端）**：基于 Spring Boot 暴露 RESTful API
- **Client（客户端 SDK）**：封装 HTTP 调用，提供 type-safe API

```
业务系统 ──HTTP──> [LightCache Server] ──调用──> [LightCache Core]
                                              │
业务系统 ──SDK ──> [LightCache Client] ──HTTP──┘
```

### 2.2 用户特征

- 后端开发者，熟悉 Java 生态
- 了解缓存基本概念（过期、淘汰）
- 可通过 curl/httpie 直接操作

## 3. 功能需求

### FR-01：键值存取

| 编号 | FR-01 |
|------|-------|
| **描述** | 系统应支持以键(String)为索引存储和读取任意 Serializable 对象 |
| **输入** | put: key(String), value(Object), ttl(long seconds) |
| **输出** | get: 命中返回 value，未命中或已过期返回 null |
| **优先级** | P0 — 核心功能 |

**验收标准：**
1. 写入 key="name" value="凯健" ttl=60，立即读取返回 "凯健"
2. 读取不存在的 key 返回 null
3. 写入相同 key 覆盖旧值

### FR-02：键删除与存在性检查

| 编号 | FR-02 |
|------|-------|
| **描述** | 支持主动删除指定键、检查键是否存在 |
| **输入** | del: key / exists: key |
| **输出** | del: void / exists: boolean |
| **优先级** | P0 |

**验收标准：**
1. 删除存在的 key 后 exists 返回 false
2. 删除不存在的 key 不抛异常

### FR-03：TTL 过期机制

| 编号 | FR-03 |
|------|-------|
| **描述** | 键到达 TTL 后自动过期，后续读取返回 null |
| **优先级** | P0 |

**子需求：**

| 子编号 | 描述 | 算法 |
|--------|------|------|
| FR-03-1 | 惰性删除：get() 时发现过期键当即删除 | 每次 get 执行 |
| FR-03-2 | 定期删除：后台线程每 10 秒随机采样 20 个键，删除其中过期的 | ScheduledExecutorService |

**验收标准：**
1. 写入 ttl=1 的键，等待 2 秒后 get 返回 null
2. 惰性删除后 size() 减 1
3. 大量过期键在定期删除周期内被清理（size 持续下降）

### FR-04：统计信息

| 编号 | FR-04 |
|------|-------|
| **描述** | 系统应提供运行期统计信息 |
| **输出** | hits(命中次数), misses(未命中次数), size(当前有效键数), hitRate(命中率) |
| **优先级** | P1 |

**验收标准：**
1. 初始 stats 各项为 0
2. 写入后 get 命中，hits=1, hitRate=1.0
3. get 不存在的 key，misses=1

### FR-05：快照持久化

| 编号 | FR-05 |
|------|-------|
| **描述** | 支持将当前全部有效键值对保存到磁盘文件，启动时自动加载 |
| **输入** | save() 手动触发 或 启动自动加载 |
| **格式** | 每行一个 JSON：`{"key":"xxx","value":"yyy","expireAt":1234567890}` |
| **优先级** | P1 |

**验收标准：**
1. 写入 3 个键后 save()，重启后 3 个键均可正常读取
2. 已过期键不写入快照
3. 空缓存 save() 生成空文件，不抛异常

### FR-06：服务端 REST API

| 编号 | FR-06 |
|------|-------|
| **描述** | 基于 Spring Boot 暴露 RESTful HTTP 接口 |
| **优先级** | P1 |

| 方法 | 路径 | 请求体 | 响应 | 说明 |
|------|------|--------|------|------|
| PUT | `/cache/{key}` | `{"value":"xxx","ttl":60}` | 200/500 | 设值/更新 |
| GET | `/cache/{key}` | — | 200 + value / 404 | 取值 |
| DELETE | `/cache/{key}` | — | 200/404 | 删除 |
| GET | `/cache/stats` | — | 200 + JSON | 统计 |
| POST | `/cache/save` | — | 200/500 | 手动持久化 |

### FR-07：客户端 SDK

| 编号 | FR-07 |
|------|-------|
| **描述** | 提供 Java 客户端 SDK，封装 HTTP 调用，API 与 CacheEngine 接口对齐 |
| **优先级** | P1（Phase 3 完成后追加） |

**验收标准：**
1. `LightCacheClient` 提供与 `CacheEngine` 相同的方法签名
2. 底层通过 HTTP 与 Server 通信
3. 连接失败抛明确的异常

## 4. 非功能需求

### 4.1 性能

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 单线程吞吐量（put） | ≥ 50,000 ops/s | JMH 微基准测试 |
| 单线程吞吐量（get） | ≥ 100,000 ops/s | JMH 微基准测试 |
| P99 延迟（get） | ≤ 1ms | 直方图统计 |

**前提条件：** JDK 17，本地 JVM，512MB 堆，1000 个键。

### 4.2 可靠性

| 编号 | 需求 | 验收标准 |
|------|------|---------|
| NFR-02-1 | 持久化数据完整性 | 快照写入过程中 crash，原有快照文件不被损坏（写临时文件 + rename） |
| NFR-02-2 | 启动容错 | 快照文件损坏时，跳过加载并记录错误日志，不阻止启动 |

### 4.3 可维护性

| 编号 | 需求 | 验收标准 |
|------|------|---------|
| NFR-03-1 | 代码结构 | 模块边界清晰，core/server/client 三层分离 |
| NFR-03-2 | 测试覆盖 | 核心模块单元测试覆盖率 ≥ 80% |
| NFR-03-3 | 日志规范 | 使用 SLF4J，INFO 级别记录启动/关闭/持久化；DEBUG 级别记录过期清理 |

### 4.4 技术约束

| 约束项 | 版本/规格 |
|--------|----------|
| JDK | 17 |
| Maven | 3.9.4 |
| Spring Boot | 3.4.3 |
| 编码字符集 | UTF-8 |
| 序列化格式 | JSON（Jackson，仅 server/client 模块引入） |
| 服务端口 | 8110（固定） |
| 依赖最小化 | core 模块零外部依赖（仅 JDK 标准库） |

## 5. 用例场景

### UC-01：典型读写场景

```
前置：服务已启动
步骤：
  1. 用户 PUT /cache/user:1001 {"value":"{\"name\":\"凯健\"}","ttl":300}
  2. 用户 GET /cache/user:1001 → 返回 JSON
  3. 等待 5 分钟后 GET /cache/user:1001 → 404
结果：TTL 过期机制正常运作
```

### UC-02：重启恢复场景

```
前置：已有 5 个有效键
步骤：
  1. 调用 POST /cache/save
  2. 停止服务
  3. 启动服务
  4. 读取 5 个键 → 均返回正确值
结果：快照持久化与启动恢复正常
```

### UC-03：高并发写入场景

```
前置：服务已启动
步骤：
  1. 10 个线程并发写入 1000 个键（随机 ttl 10~60 秒）
  2. 过程中持续调用 GET /cache/stats
  3. 验证无 ConcurrentModificationException、无数据错乱
结果：并发安全，stats.size 与预期一致
```
